Week 1 – Programming for Data Analysis: Mindset & Warm-Up

CSC1184 is a second-year module about using Python to work with data: loading it, cleaning it, exploring it, and building simple models. In this first week we focus on language and mindset rather than heavy code: what counts as “data”, what kinds of questions we can ask, and how the Python you learned earlier fits into that picture.

By the end of this lecture, you should be able to:
  • Explain what “programming for data analysis” means in plain language.
  • Use terms like dataset, observation, and variable correctly.
  • Recognise a simple workflow from question → data → conclusion.
  • Describe where Python and its data libraries fit into that workflow.
  • Read and modify small Python examples that analyse lists of numbers.
Big idea In this module, code is not the “final product”. The real goal is the insight you get from the data and how clearly you can explain it to someone else.

1. What Is Programming for Data Analysis?

In first-year programming you often wrote code to solve small, self-contained problems: tax calculators, guessing games, simple simulations. In this module, the problems start with data. You have measurements, logs, survey responses, or records, and you want to learn something from them.

You can think of “programming for data analysis” as sitting at the intersection of:

  • Programming – writing correct, readable Python code.
  • Statistics – reasoning about variability, averages, relationships.
  • Domain questions – what you actually care about (students, finance, sports, etc.).

A data-analysis task might be as simple as “What is the typical exam score?” or as ambitious as “Can we predict who will pass?” Either way, your Python skills are there to help you move from raw data to a clear, evidence-based answer.

Earlier programming modules
  • Focus on language features and basic problem solving.
  • Work with small, carefully chosen examples.
  • Success = program matches the specification exactly.
Programming for data analysis
  • Focus on making sense of real or realistic datasets.
  • Data can be messy, incomplete, or noisy.
  • Success = you can defend what the data suggests (and what it does not say).

2. Datasets, Variables & Observations

Most of the time, we will picture data as a table, similar to a spreadsheet:

  • A dataset is the entire table – all rows and all columns together.
  • An observation (or record) is one row – one student, one match, one transaction, one day.
  • A variable is one column – something we measure or record about each observation (age, score, programme, price, category, date).

2.1 Types of variables

Different types of variables call for different plots, summaries, and models. For this module, it is enough to recognise three broad types:

  • Numeric – things you can sensibly add and average (age, score, height, price).
  • Categorical – labels or groups (programme, country, grade band, product category).
  • Time-based – dates and times (exam date, timestamp of a log entry).
Activity – Spot observations and variables Think of a dataset you might actually see in your degree: module marks, lab attendance, or a small log of system requests. What is one observation in that dataset? List three variables you could record about it.

3. Kinds of Questions We Ask About Data

Not all data-analysis questions are the same. It helps to name a few common types, both so you can frame your own questions and so you can recognise what a client or colleague is really asking for.

  • Descriptive – “What does this dataset look like?” Examples: distribution of exam marks, number of students per programme, average daily temperature over a month.
  • Comparative – “Are these groups different?” Examples: average score in two different modules, sales before vs after a promotion.
  • Relational – “Are these variables related?” Examples: relationship between lab attendance and exam score; price vs size for houses.
  • Predictive – “Can we predict one thing from others?” Examples: predict exam score from coursework and attendance; predict whether a customer will buy again.

In this module we will mainly tackle descriptive, comparative, and relational questions, with a gentle introduction to predictive models later in the semester.

4. A Simple Data Analysis Workflow

Many data-science workflows follow a similar pattern: start from a question, obtain data, clean it, explore it, build a model, and communicate results. The exact diagram differs between sources, but the core pattern is the same.

→
Obtain
→
Clean
→
Explore
→
Model
→
Communicate

4.1 The stages in more detail

Ask

Clarify what you want to know. “Is attendance related to marks?” is clearer and more testable than “Is this dataset interesting?”. A good question mentions the variables you care about and the kind of comparison or pattern you are interested in.

Obtain

Get data from somewhere: CSV exports, databases, log files, APIs, manual entry. In this module we mainly use CSV files and small text files, but the same ideas extend to larger systems. You should always ask where the data came from and how it was collected.

Clean

Fix obvious problems: missing values, wrong types, duplicated rows, inconsistent labels (for example, “CS”, “Computer Science”, and “CompSci” being used for the same programme). Cleaning is often the longest part of a project and has a big impact on the quality of your conclusions.

Explore

Compute summaries and create plots to understand distributions, patterns, and anomalies. You might look at averages, histograms, boxplots, or simple scatterplots. The goal here is curiosity: get a feel for the data before committing to any one model or story.

Model

Fit a simple model to capture a relationship or make basic predictions. In this module that might be a linear regression or a simple classifier. A model should be interpretable and checked against the data; it is not a magic answer generator.

Communicate

Turn your work into something another person can read or act on: a short report, a slide deck, or a notebook with commentary. Good communication explains what you did, what you found, and what the limitations are, rather than just dumping code and numbers.

The rest of the module essentially walks this workflow with increasing levels of sophistication. Week 1 sits mainly in “Ask” and “Explore” for very small datasets.

5. Why Python? Where This Module Fits

Python is widely used in data analysis because of the ecosystem around it: NumPy for efficient arrays, pandas for data tables, Matplotlib and other libraries for plots, scikit-learn for models, and so on. This module does not try to cover everything; it focuses on the ideas and tools you are likely to see again.

CSC1184 aims to:

  • Strengthen your basic Python when applied to data rather than toy puzzles.
  • Introduce core libraries (NumPy, pandas, basic plotting) at a comfortable pace.
  • Give you a solid mental model of the data-analysis workflow.
  • Prepare you to read and adapt examples from standard Python data-analysis resources.

In Week 1 and Week 2 we stay close to “plain” Python and small tables so that when we add libraries, they feel like natural extensions rather than magic.

6. Python Refresh: Lists & Built-ins

Before we look at the code examples, we slow down and name exactly what kinds of Python objects we are using and what the basic operations mean. In Week 1 we only need:

  • Numbers (integers and floats).
  • Lists (ordered collections of values).
  • Built-in functions like len, sum, min, max, and print.
  • Simple loops with for.

6.1 What is a list?

A list in Python is an ordered collection of items. Important properties:

  • Written with square brackets, for example: [55, 62, 71, 48].
  • The order matters: the first element is at position 0, the second at 1, and so on.
  • You can access a single element by its index, e.g. scores[0] is the first score, scores[1] the second.
  • Lists are mutable: after creating a list, you can change, add, or remove elements (for example, scores.append(100) to add a value).
  • Lists can grow or shrink: you are not stuck with a fixed size.
  • Technically, a list can contain values of different types, but in data analysis we usually keep one list to “one meaning” (for example, all exam scores, all temperatures, or all daily study times).
