Week 1 Lab – From Algorithms to First Scripts

In this first lab you will not learn any new maths or fancy Python features. Instead, you will: (1) get comfortable with the online environment and (2) see how your everyday algorithms can turn into a very simple script that the computer follows literally.

Lab goals
  • Become comfortable running and resetting Python code in the browser.
  • Connect the idea of an algorithm to a sequence of printed steps.
  • See how the computer follows instructions exactly as written.
  • Get a first, gentle introduction to the print() function.
  • Get a first, gentle introduction to what an error is in Python.
Important Today is still about concepts. We will only use one Python command: print(...). We will name errors, but not go into technical detail yet.

2. Everyday Algorithms (No Code)

We begin off-screen, without any code. This mirrors the lecture: you design an algorithm first, and only then think about how a computer might follow it.

2.1 Choose a task

(5–10 minutes)

Pick one of these tasks (or invent a similar everyday one):

  • Making a cup of tea or coffee.
  • Locking up your house when you leave.
  • Ordering food using a delivery app.
  • Getting from your front door to your classroom.

2.2 Write the algorithm

  1. Write down:
    • The input (what things you start with).
    • The goal/output (what counts as “done”).
  2. List 8–12 steps to get from input to output.
  3. Make each step a simple, concrete action:
    • Avoid “do it properly”, “some” or “a bit”.
    • Prefer: “Fill the kettle with water up to the line”.
Programming link What you are writing here is already an algorithm. In a few weeks, you’ll be able to translate this sort of description into Python, using variables, conditions, and loops. Today we will just turn it into a text-based script.

2.3 Second algorithm (more decisions)

(10–15 minutes)

Write a second algorithm that has at least one clear decision point (an “if … then … else …” moment). For example:

  • “If the app cannot find a driver, then…”
  • “If the door is already locked, then…”
  • “If the bus is full, then…”
  1. Write the input and output again for this new algorithm.
  2. Identify at least one step where the procedure can branch: e.g. “if X happens do A, otherwise do B”.
  3. Write out both branches explicitly as separate steps. (We still won’t code decisions yet, but it’s good to notice them.)

3. From Algorithm to Script

Now we’ll make the computer display the steps of your algorithm one by one. The computer is not doing the steps yet – it’s just printing them. But that is enough to see the connection between “step-by-step description” and “program”.

3.1 Tiny introduction to print()

In Python, the basic command we’ll use today is print(). You can read it as:

What does print("Hello") mean?
  • print – the name of a built-in command (a function).
  • ( ... ) – round brackets that surround the thing we want to show.
  • "Hello" – text between quotation marks. Later we’ll call this a string.
When Python sees print("Hello"), it displays the word Hello in the output area.

For this lab, you can think of print("some text") as: “tell the computer to show this line of text”.

5. First Look at Errors

        

          When Python cannot understand your instructions, it stops and shows an
          error message. That doesn’t mean you’re bad at programming –
          it just means the computer needs you to fix something.
        


        
5.1 What is a syntax error?

        

          The first kind of error you will see is usually a SyntaxError.
          “Syntax” is just a word for the rules of how code is written
          (where brackets and quotes go, how lines are structured, etc.).
        

        

          SyntaxError
          A SyntaxError happens when the structure of the code is broken,
          so Python can’t even start running your instructions. Examples:
          
    
            
  • Missing closing " or ).
    • 
            
  • Using Print() instead of print().
    • 
            
  • Random extra characters that don’t belong.
    • 
          

        

        

          In later weeks you will see other kinds of errors too, but today we only care about this
          first, common one: syntax errors.
